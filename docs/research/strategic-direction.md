## What Nexa Is

Nexa is the only compliance-aware compute placement system for Kubernetes. It answers a question no other tool answers: **"prove that this sensitive workload ran on infrastructure that meets these specific compliance requirements."**

Batch schedulers (Kueue, Volcano, YuniKorn) solve resource efficiency. Policy engines (OPA Gatekeeper, Kyverno) solve admission control. Nexa solves compliance-aware placement — ensuring pods land on nodes that satisfy privacy, data residency, and confidential compute policies, with an auditable trail and dynamic contamination tracking.

After 18 sprints, Nexa has: 4 scheduler plugins (Region, Privacy, Confidential, Audit), a node state controller with remote attestation, a validating webhook, a compliance report CLI, Kueue integration, Helm charts, and 137 unit tests + 22 smoke tests at ~93% coverage.

## Who Needs This

### Primary Markets

**Financial Services** — Highest fit. Banks spend billions on AI infra (JPMorgan $2B/year, BofA $4B, Goldman $3B) and face strict data isolation mandates from OCC, SEC, FCA, MAS. They currently solve this with dedicated clusters per business unit (expensive) and manual compliance audits (error-prone). Nexa's org isolation + audit trail is exactly what regulators are starting to demand. DevOps investment in banking is growing 20-25% annually.

**Sovereign Cloud / EU Data Residency** — Direct product-market fit. The sovereign cloud market is projected at $154B in 2025, growing to $823B by 2032. GDPR, Schrems II, and the EU AI Act require provable region-specific execution. Nexa's region plugin with policy-driven enforcement and anti-spoofing webhook maps directly to this requirement. AWS ($7.8B), SAP ($20B+), and European providers are all building sovereign infrastructure that needs this scheduling layer.

**Healthcare** — High fit. HIPAA's January 2025 Security Rule update made all safeguards mandatory. Multi-hospital systems sharing clusters need provable workload isolation at the compute level, not just network/storage. Current solutions overprovision at 3-5x cost.

### Secondary Markets

**Defense & Intelligence** — High technical fit, slow go-to-market. FedRAMP/ITAR procurement cycles are long, but the need for classification-level isolation with audit trails on shared clusters is acute. Platform One (DoD k8s) lacks privacy-aware scheduling.

**AI/ML Platform Providers** — Force multiplier. CoreWeave, Lambda, Modal, Anyscale all run multi-tenant GPU infrastructure and need to prove isolation to enterprise customers. One integration reaches thousands of end users. 84.1% of AI infra spending is in shared/cloud environments.

### Opportunity Size

- SAM (Kubernetes + multi-tenant + compliance): $5-17B by 2026, $13-39B by 2030
- Realistic open-source infrastructure capture: $50M-$200M ARR at maturity
- Real leverage: becoming the compliance audit backbone — the organizational dependency on Nexa's audit output is the moat, not the scheduler itself

## What's Holding Nexa Back

### Architecture Limits at Scale

The current architecture works at small-to-medium scale (< 500 nodes, < 10K pods) but has specific bottlenecks that become blockers at production scale for the target customers.

**Immediate (any scale):**
- Policy re-parsed from JSON on every Filter/Score call — 8+ parse cycles per pod per scheduling cycle
- `podLister.List(labels.Everything())` scans all cluster pods on every node reconciliation
- Singleton provider failure is permanent — one init error bricks the scheduler until restart

**Medium scale (100-500 nodes):**
- No leader election on controllers — HA deployment causes label patch races
- MergePatch without resourceVersion — concurrent patches silently overwrite each other
- Sequential attestation verification — O(TEE-nodes × latency) per tick

**Large scale (500+ nodes):**
- Eventual consistency race between scheduler and controller — pods can land on unwiped nodes
- Node labels as sole state medium — no history, no per-device granularity, 256KB total limit
- Single scheduler instance — no sharding for throughput

### GPU Blind Spot

Nexa's trust boundary ends at CPU and disk. For the target market (regulated AI workloads), this is a critical gap:

- GPU VRAM sits outside TEE protection — confidential compute claims are misleading for GPU workloads
- No GPU memory wipe in the pod lifecycle — `nexa.io/wiped=true` doesn't cover VRAM
- Org isolation checks miss GPU sharing (MPS, MIG) — node-level checks pass while device-level isolation is violated
- Compliance reports generated by `cmd/compliance/` overstate protection for GPU workloads

### Kubernetes Lock-in

The highest-value AI training workloads run on Slurm, not Kubernetes. Nexa's policy engine is embedded in k8s scheduler plugin code with direct informer dependencies. Extending compliance-aware scheduling to Slurm or YARN requires extracting the policy engine into a standalone service — a healthy architectural change that the scale issues also demand.

## Where to Go Next

### Phase 1: Production Hardening (2-3 sprints)

Fix the architecture issues that would embarrass Nexa in a production deployment. These are no-tradeoff improvements.

| Work Item | Why | Effort |
|---|---|---|
| Cache parsed policy via `atomic.Pointer`, invalidate on informer events | Eliminates 8+ JSON parse cycles per pod. Prerequisite for any scale. | S |
| Add `spec.nodeName` informer index to controller | Prevents O(all-pods) scan. Trivial, no reason to ship without it. | S |
| Retry/backoff on provider singleton init | One transient failure shouldn't permanently brick the scheduler. | S |
| Leader election for node state + attestation controllers | Required before anyone runs HA. Without it, label patches race. | M |
| Switch to Server-Side Apply for node patches | Eliminates silent overwrites from concurrent patches. | M |
| Parallel attestation via `errgroup.SetLimit` | Sequential verification doesn't scale past 50 TEE nodes. | S |

This phase makes Nexa credible for medium-scale production deployment (100-500 nodes). It's the minimum bar for the first enterprise customer.

### Phase 2: GPU Compliance (1-2 sprints)

Close the GPU trust gap with the simplest viable approach: **node-level GPU exclusivity**. Don't try to solve per-device isolation — that's a problem most customers avoid by dedicating GPU nodes to tenants.

| Work Item | Why | Effort |
|---|---|---|
| GPU capability labels (`gpu-present`, `gpu-type`, `gpu-confidential-compute`) | Acknowledge GPUs exist in the compliance model. | S |
| Filter check: if pod requests GPU + `privacy: high`, require `gpu-confidential-compute=true` | Prevent misleading confidential compute claims. | S |
| Policy flags: `requireConfidentialGPU`, `gpuWipeRequired` | Give operators control over GPU compliance policy. | S |
| GPU exclusivity mode: strict org isolation on GPU nodes prevents cross-tenant sharing | Reduces GPU isolation to node isolation, which Nexa already handles. | S |
| Update compliance report to flag GPU workloads and their coverage gaps | Honest audit trail — don't claim protections that don't exist. | S |

This phase makes Nexa's compliance story honest for GPU workloads. It's a market requirement — financial services and AI platform providers run GPU clusters, and Nexa's current claims don't hold there.

### Phase 3: Policy Service Extraction (2-3 sprints)

Extract the policy engine into a standalone gRPC service. This is the architectural pivot that unlocks both scale and multi-scheduler support.

```
┌─────────────────────────────────────────────────┐
│                 Nexa Policy Service              │
│  (gRPC — policy evaluation, node state store,   │
│   audit logging, compliance reporting)           │
└──────────┬──────────┬──────────┬────────────────┘
           │          │          │
     ┌─────┴───┐ ┌────┴────┐ ┌──┴──────────┐
     │ k8s     │ │ Slurm   │ │ YARN        │
     │ plugins │ │ plugins │ │ plugins     │
     │ (Go)    │ │ (C/sh)  │ │ (Java)      │
     └─────────┘ └─────────┘ └─────────────┘
```

| Work Item | Why | Effort |
|---|---|---|
| `cmd/nexa-policy-server/` with gRPC API: `EvaluateNode()`, `RecordDecision()`, `GetNodeState()` | Transport-agnostic policy evaluation. Required for Slurm, good for k8s scale (single cached policy instance). | L |
| Refactor k8s plugins as thin gRPC clients | Existing plugins call the policy service instead of importing `pkg/policy` directly. | M |
| Node state store abstraction (k8s CRD, SQLite, or embedded etcd backend) | Required for non-k8s schedulers and for per-device GPU state at Tier 4 scale. | M |
| Audit event ingestion API | Any scheduler integration can emit compliance events in a common format. | S |

**Why now, not later**: The policy extraction fixes the Tier 1 scale issue (policy caching) and the Tier 4 scale issue (node state as CRD) simultaneously. It's the architectural foundation for everything that follows. If Nexa ever needs Slurm support (and the market analysis says the highest-value AI training runs on Slurm), this refactoring is the prerequisite. Better to do it before the k8s-specific code calcifies further.

### Phase 4: First Customer Integration (demand-driven)

These are not speculative — they should be triggered by specific customer needs.

**If a Slurm customer appears:**
- Slurm `job_submit` plugin (C) that calls the policy service
- Prolog/epilog scripts for node contamination tracking
- GPU agent DaemonSet for VRAM wipe on GPU nodes
- Effort: 2-3 sprints on top of the policy service

**If scale demands it (500+ nodes):**
- PreFilter gate for real-time wipe state verification
- Per-cycle policy snapshot via CycleState
- Async audit pipeline (bounded channel + background writer)
- Scheduler sharding by namespace or workload class
- Effort: 2-4 sprints

**If GPU device-level isolation is demanded:**
- NexaNodeState CRD with per-device state
- DRA integration (k8s 1.31+) or Reserve plugin for device-level claims
- GPU agent DaemonSet with VRAM wipe verification
- Kueue capacity feedback loop (effective GPU count per org)
- Effort: 5-8 sprints, significant k8s API surface risk

**If YARN is needed:**
- YARN node label bridge driven by policy service
- Capacity Scheduler extension (Java) for compliance-aware allocation
- Effort: 2-4 sprints, requires Java expertise

## What NOT to Build

**GPU topology / gang scheduling / preemption.** Kueue, Volcano, and KAI Scheduler solve these. Building inferior versions dilutes focus. The roadmap's original decision was correct — stay complementary, not competitive.

**A platform / control plane / UI.** Nexa's value is as an infrastructure primitive. Adding a management layer makes it competitive with Rafay, D2iQ, and cloud provider consoles. Stay in the scheduling/compliance layer.

**Per-device GPU isolation before a customer asks.** Node-level GPU exclusivity covers the security-sensitive deployment model. Per-device isolation (MIG-aware, DRA-integrated) is architecturally interesting but solves a problem most target customers avoid by dedicating GPU nodes. Build it when demand proves it's needed.

**YARN integration speculatively.** The long-term trend is YARN-to-k8s migration via YuniKorn. Unless a specific customer runs YARN with compliance needs, let convergence handle this.

## Risks to the Strategy

| Risk | Mitigation |
|---|---|
| "Good enough" alternatives — enterprises accept namespace isolation + manual audits | Focus on industries where regulators explicitly require compute-level evidence (HIPAA 2025, EU AI Act, DORA) |
| Cloud provider bundling — AWS/GCP/Azure build compliance scheduling natively | Stay cloud-agnostic and open-source. Nexa works across providers; their solutions won't. |
| Kueue adds compliance features | Deepen the integration (co-scheduling protocol, capacity feedback). If Kueue subsumes Nexa, contribute upstream. |
| Regulatory lag — regulators don't mandate compute-level isolation evidence | The trend is favorable but not guaranteed. Hedge by making the compliance audit story valuable even without mandates (cost reduction from shared clusters). |
| Slurm market shrinks faster than expected | The policy service extraction is valuable for k8s scale alone. Slurm support is additive, not foundational. |

          │

  ├── K8s plugins as thin clients
  ├── Node state store abstraction
  └── Audit event API
          │
Phase 4: Customer-Driven               [on demand]
  ├── Slurm integration (if customer)
